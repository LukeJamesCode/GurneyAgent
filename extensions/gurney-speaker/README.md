# gurney-speaker

Handmade smart-speaker front-end for Gurney. The extension hosts a WebSocket server that ESP32-S3 "Gurney puck" devices connect to: they stream microphone audio in (raw 16 kHz mono PCM after on-device wake-word detection or a push-to-talk button) and play TTS replies back (OGG/Opus). Display state (`idle | listening | thinking | speaking | muted`) and volume/mute updates ride the same connection.

It depends on **gurney-voice** for whisper.cpp STT and Piper TTS — no duplicated audio plumbing.

> **Status:** server side feature-complete. WS server, session machine, STT, orchestrator dispatch, device-state persistence, and TTS are all wired and unit-tested. Firmware boots end-to-end on hardware (WiFi, WS, buttons, PTT); the remaining firmware work is bench-bringup of the panel and the Opus decoder — see `firmware/gurney-speaker/README.md`.
>
> **Dispatch:** when `owner_chat_id` is set, the device's turns go through Gurney's orchestrator — same conversation history as Telegram, same tools (calendar, reminders, weather, briefings, etc.), same hallucination guard. Replies get post-processed through `shortenForSpeech` so markdown and long monologues don't end up in the TTS stream. Without `owner_chat_id` the device falls back to a stateless `host.llm` chat (no tools, no history) — useful for keeping a device isolated.

## Hardware target

- ESP32-S3 dev board
- 2× I2S omni mics (e.g. INMP441 / ICS-43434)
- MAX98357A 3 W class-D amp + speaker pulled from a gen 3 Echo Dot
- 1.28" round GC9A01 LCD over SPI
- 4 buttons: volume +, volume −, mute (hard kill of mic + TTS), one reserved for pairing

## Wire protocol (binary WS frames, opcode in byte 0)

| Op   | Direction | Payload                                            |
|------|-----------|----------------------------------------------------|
| 0x01 | C→S       | JSON `{deviceId, secret, fwVersion}` (hello)       |
| 0x02 | S→C       | JSON `{ok, displayStyle, volume, muted, voiceId}`  |
| 0x10 | C→S       | wake fired (empty)                                 |
| 0x11 | C→S       | raw int16 PCM @ 16 kHz mono (~20 ms frames)        |
| 0x12 | C→S       | end-of-utterance hint (empty)                      |
| 0x20 | S→C       | JSON `{state}` — display update                    |
| 0x21 | S→C       | OGG/Opus bytes (chunked TTS)                       |
| 0x22 | S→C       | end-of-TTS marker (empty)                          |
| 0x30 | C↔S       | JSON `{volume?, muted?}` — state sync              |
| 0x31 | C→S       | JSON `{button}` — physical button event            |
| 0x7F | S↔C       | keepalive (empty)                                  |

## Settings

See `settings.schema.json`. Notable ones:

- `device_shared_secret` — generated by setup; flash into the firmware NVS image.
- `display_style` — `minimal` or `orb`; pushed to the device in the welcome frame.
- `owner_chat_id` — device turns post into this chat's conversation history. When set, the orchestrator path is used (full tools + memory). When unset, the device falls back to a stateless LLM chat.
- `vad_silence_ms` / `max_utterance_sec` — server-side end-of-turn detection.
- `whisper_model_path` / `voice_model_path` — point at the artifacts gurney-voice already downloaded (typically `~/.gurney/extension_state/gurney-voice/whisper-models/ggml-base.en.bin` and the matching Piper `.onnx`). This avoids a second download.
- `speech_max_chars` — hard cap on the spoken reply length. Replies longer than this get clipped at the nearest sentence boundary; 600 chars ≈ 30 seconds spoken.
- `llm_profile` / `llm_max_tokens` / `system_prompt` — model behaviour for the **direct LLM fallback only**. The orchestrator path uses Gurney's main system prompt + the extension prompt fragments. Keep replies short — voice latency punishes long answers.

## Persisted state

Per device, the extension keeps a row in the `speaker_devices` table tracking `last_seen`, `last_volume`, and `muted`. On reconnect, the welcome frame replays the device's last-known volume / mute instead of snapping back to the schema defaults. This means a power-cycled puck doesn't surprise you by booting back to 60 % volume.
